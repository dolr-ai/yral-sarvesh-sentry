# SECURITY.md — threat model + secrets inventory for yral-sarvesh-sentry

> **Status (2026-04-21):** stub. Will be written in **Phase 9** of the rollout (see `PROGRESS.md`). The notes below are the starting points, not the final model.

## What this document will cover (Phase 9)

1. **Trust boundary map** — who can reach Sentry, from where, under what auth.
2. **Secrets inventory** — every secret Sentry depends on, where it lives, how to rotate it.
3. **Public attack surface** — everything reachable from the internet, with mitigations.
4. **Internal attack surface** — everything reachable from sarvesh-1/sarvesh-2 (the Caddy hosts) but not from the internet.
5. **Data classification** — what PII ends up in Sentry (stack traces, request params, breadcrumbs), and what we do about it.
6. **Incident response** — what to do if the Sentry Postgres dump leaks, if the Google OAuth secret leaks, if a `@gobazzinga.io` account is compromised.

## Initial starting points (to be finalized in Phase 9)

### Public surface

- `https://sentry.sarvesh.yral.com` on sarvesh-1 + sarvesh-2 (via Caddy, TLS terminated by Caddy, wildcard Cloudflare DNS).
- Sentry's internal nginx on sarvesh-3 is bound to `127.0.0.1:9000` — NOT reachable from the internet.
- Caddy reverse-proxies over either the Docker Swarm `web` overlay or a private-IP route to sarvesh-3 (connectivity path is a Phase 3 decision).

### Auth posture

- **Primary**: Google Workspace SSO. Domain whitelist enforced in `GOOGLE_DOMAIN_WHITELIST = ['gobazzinga.io']`.
- **Fallback**: one local superuser account (Sarvesh's), TOTP-protected, retained indefinitely so that OAuth misconfig cannot lock us out.
- **Public signup disabled** via `auth.allow-registration: false` in `sentry/config.yml`.
- **Require SSO** enabled in Organization Settings → Auth after Phase 4.

### Secrets this service depends on

Initial list — to be confirmed once Phase 2 is done:

| Secret | Where it lives | Rotation trigger |
|---|---|---|
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | GitHub Secrets on this repo + `sentry/config.yml` on sarvesh-3 | Annual or on-leak |
| Sentry's own internal secret key (`system.secret-key`) | Auto-generated by `install.sh` into `sentry/config.yml` on sarvesh-3 | On compromise only — rotating invalidates all sessions |
| Postgres password for Sentry's own DB | Auto-generated by `install.sh` into `.env` on sarvesh-3 | On compromise only |
| Sarvesh's local superuser password | In Sarvesh's password manager | On compromise only |

### Things that are intentionally NOT secrets

- `SENTRY_DSN` values — they're public identifiers that let the SDK send events. They do not let a holder READ events, only WRITE them. Treat as "somewhat-public."
- The URL `sentry.sarvesh.yral.com` — public.
- Sentry version (`26.6.0`) — public.

### Open question to resolve in Phase 9

- Do we want to enable Sentry's built-in feature for **scrubbing PII from stack traces** (sensitive field names like `password`, `api_key`)? Template and chat-ai should have `send_default_pii=False` after Phase 6, but server-side scrubbing is a belt-and-braces layer.

## Until Phase 9 ships

Basic posture is: the install procedure sets sensible defaults. Don't commit any `.env` file to git (enforced by `.gitignore`). Don't expose port 9000 on sarvesh-3 to anything but loopback. Do not disable the superuser fallback.
